01

Architecture Concerns

Critical The Express Microservice is a Time Bomb
Running a standalone Express server to bypass Vercel timeouts is clever engineering — but it's an unmanaged, stateful process. When it crashes, all WebSocket connections die silently, active research jobs are lost, and no client will know why. There is no supervisor, no restart policy, no health check endpoint, and likely no PM2 or container orchestration.
Run this on Railway/Fly.io in a Docker container with PM2 or a systemd supervisor. Expose a /health endpoint. Add a Redis-backed job queue (BullMQ) so jobs survive restarts. Inngest already handles retries — lean into it fully.
Critical No Row-Level Security on NeonDB
Org isolation is enforced entirely in application code via organizationId filters on Prisma queries. A single missed WHERE clause leaks every org's data. This is a GDPR/SOC2 nightmare and a common attack vector. Stripe, Linear, and Notion all learned this the hard way early.
Enable PostgreSQL Row-Level Security (RLS) on NeonDB. Create a policy: USING (organization_id = current_setting('app.org_id')). Set the config before every transaction in a Prisma middleware. Defense in depth.
Warning WebSocket Rooms Scale to ~1 Server
Socket.io with in-memory rooms works beautifully on one server. The moment you need two instances for redundancy or load, users on Server A can't see events from users on Server B. This isn't a distant problem — a single deploy restart causes this.
Add socket.io-redis-adapter with an Upstash Redis instance. 3 lines of code. Solves horizontal scaling forever. Worth doing now before any production load.
Warning Inngest Job Fan-out Has No Backpressure
If 50 orgs trigger research jobs simultaneously, 50 headless Puppeteer browsers spin up at once. This will OOM the Express server and cause all 50 to fail. There's no concurrency limit, no queue depth awareness, and no job priority system.
Set Inngest concurrency limits per org (e.g. max 2 concurrent research jobs per org). Add a job_queued state in the DB so users see "Your research is queued (position 3)" — this is actually a delightful UX, not a limitation.
Opportunity tRPC + Prisma is Excellent — Go All-in
The tRPC + Prisma + TypeScript stack is genuinely what Linear, Vercel, and Planetscale use internally. This is a strength. But the mixing of tRPC with direct Server Actions suggests inconsistency in the codebase. Pick one mutation pattern and stick to it, or you'll have a split-brain debugging experience in 6 months.
Standardize on tRPC for all mutations. Use Server Actions only for form-based progressive enhancement (rare). Add Zod validation on every tRPC input. The type-safety payoff compounds exponentially as the codebase grows.
02

Performance & Reliability

Critical Zero Observability = Flying Blind in Production
There is no mention of error tracking (Sentry), distributed tracing (OpenTelemetry), or structured logging. When the research engine silently fails at 2 AM, you won't know until a customer tweets at you. Google/Meta engineers get paged within 60 seconds of an error spike. You'll find out in 3 days.
Add Sentry (free tier handles plenty). Add Pino or Winston with structured JSON logging to your Express server. Add a /metrics endpoint for Prometheus if you ever self-host. Costs $0 and saves days of debugging.
Warning Infinite Canvas Will Lag at Scale
DOM-based canvas with absolute positioning works up to ~200-500 nodes. Beyond that, layout thrashing, expensive reflows, and React reconciliation overhead will cause noticeable jank. Figma uses WebGL. Miro uses a hybrid virtualization approach. This isn't a day-1 problem, but it's an architectural ceiling.
Implement spatial indexing (quadtree or R-tree) to only render nodes within the viewport. Virtualize off-screen nodes. This single optimization extends the DOM approach 10x before needing a WebGL rewrite.
Warning Prisma N+1 Queries on Dashboards
The Team Hub aggregates metrics from Projects, Tasks, Clients, Goals, and Activity. Without explicit include optimization and Prisma query logging turned on, dashboards likely fire 20-50 individual DB queries per load. NeonDB on free tier has 0.5s cold start latency — this compounds badly.
Enable Prisma query logging in development. Use prisma.$transaction for atomic reads. Add a Redis cache layer (Upstash) for dashboard aggregations with a 30-second TTL. The dashboard doesn't need to be real-time for static metrics.
03

Developer Experience

Warning No E2E Test Coverage Mentioned
A platform touching CRM data, financial records (lifetimeValue, outstandingBalance), and authenticated multi-tenant data with zero mention of Playwright or Cypress tests is a confidence problem for enterprise buyers. When a CRM deal accidentally shows in the wrong org, you need to know before they do.
Add Playwright tests for critical paths: auth flow, deal CRUD, canvas save/load, research job creation. 10 tests covering happy paths. Run on every PR. This is non-negotiable for B2B SaaS trust.
Opportunity Zustand + Undo Stack is Well-Architected
The canvas undo/redo with a 50-step history stack in Zustand is genuinely good engineering. This is exactly what Figma's internal engineers recommend for canvas state. Consider extracting this pattern into a useHistoryStore generic hook — it could power task editing, pipeline stage changes, and goal updates too.
Opportunity AI Integration is Scattered — Centralize It
Gemini and Groq are used across research, text generation, meeting notes, and day planning — but each call is likely its own implementation. No prompt versioning, no usage tracking, no fallback logic. When Groq's API is down, which features silently break?
Create an ai-gateway.ts module that wraps all LLM calls. Add: provider fallback chain (try Groq → fallback Gemini), prompt templates as constants (version-controlled), usage logging to DB, and per-org token budgets for cost control.

05

What's Missing That Competitors Have

Gap No Offline / Optimistic UI
Every mutation requires a network round trip. Linear's killer feature is that it feels instant — it writes to local state first, syncs in background. On a bad connection (conference WiFi, mobile), this app will feel broken while Linear feels native. This is the #1 differentiator people don't realize they care about until they feel it.
Implement optimistic updates in tRPC with rollback on error. Start with task completion — the most common action. TanStack Query makes this trivial with onMutate + onError rollback.
Gap No Public API / Webhooks for Customers
Enterprise customers need to connect this to their existing stack. No REST API, no webhook system, no Zapier integration means you're a dead-end tool. Every competing platform (Notion, Monday, ClickUp, Pipedrive) solved this in year 1. Without it, you cannot land mid-market deals.
Build a public REST API alongside tRPC (just expose tRPC procedures as REST). Add a WebhooksConfig table: customers register URLs, and your Inngest jobs fire them on events. This unlocks Zapier natively and eliminates 80% of integration objections.
Gap No Native Mobile App or PWA
The well-being tracker, daily planning, and task completion — the highest-frequency features — are things users want to do from their phone. A PWA with service worker caching would unlock 2x daily active usage. No mobile = no habit formation = churn.
Gap No Notification / Alerting System
Tasks go overdue. Deal stages change. Research completes. None of these trigger push notifications or in-app alerts. The activity feed shows what happened — but only if you're already looking at the dashboard. The app has no way to pull users back in, which is a retention and awareness problem.
Build a Notification entity in Prisma. Trigger notifications via Inngest on key events. Render an inbox (bell icon) in the nav. Add email digest (Resend is already integrated). This closes the engagement loop.
Untapped The AI Research Engine is a Standalone Product
The Puppeteer-driven autonomous research engine with real-time terminal streaming is genuinely novel and valuable on its own. Most CRMs charge $500+/mo for basic enrichment that doesn't do what this does. This deserves its own pricing tier, its own landing page, and its own API product. Don't bury it.
