02Research Engine

The @google/genai Migration is the Right Infrastructure Move

Migrating from the legacy SDK to the new @google/genai Interactions API is not just a library upgrade — it unlocks native streaming reliability, improved function calling, and access to Gemini's latest capabilities. This is the correct foundation for where you need to take the research engine next.

Strength Search Grounding is a Fundamental Quality Upgrade
Integrating Google Search retrieval directly into the AI agent — rather than using Puppeteer as the only data source — is a significant architectural improvement. This gives the agent access to indexed, ranked, up-to-date web content without the fragility of headless browser navigation. Grounded generation means the model cites real sources rather than hallucinating plausible-sounding ones. The combination of Puppeteer for deep arbitrary-page reading AND Google Search for broad indexed retrieval is actually a stronger architecture than either alone.
Strength Full-Page Source Viewer is the Right UX for Trust
Replacing modals with a full-page, markdown-rich source viewer is a product maturity decision. Users need to be able to verify what the AI read before they act on the research. This is the difference between a tool that creates dependency and one that builds trust. It's also the feature that makes the product credible to skeptical enterprise buyers who want to audit outputs before sharing them with clients.
P1 · Architecture Grounding + Better Streaming ≠ Multi-Agent Research
The SDK migration and search grounding improve the quality and reliability of what the agent produces. But the agent is still one LLM call doing everything serially. Grounded generation with a single context window still hits the same fundamental bottleneck: the more thorough the research, the longer the user waits. Google's own deep research implementation uses parallel subagents specifically to solve this. You've upgraded the engine — you still need to upgrade the architecture.
Next step → The search grounding integration you've built is actually the subagent output layer. The missing piece above it is an Orchestrator that: (1) decomposes the query into sub-questions, (2) spawns parallel Gemini calls with search grounding for each sub-question, (3) aggregates and synthesises the results. This tripling of architecture gives you reports in one-third the time with three times the depth.
P1 · Quality Source Citation Without Credibility Scoring is Half a Solution
The agent now cites real sources from Google Search — which is meaningfully better than uncited outputs. But all cited sources are currently treated with equal weight. A Wikipedia edit from yesterday and a peer-reviewed market study from a reputable research firm will both appear as citations with identical visual weight. For enterprise users making business decisions from research reports, this is a trust problem: they can't tell good intelligence from noise.
Next step → Add a credibility_score (1–10) to each citation computed from: domain authority tier (academic/government > industry publication > news > aggregator > forum), publication date recency, and cross-source corroboration count. Surface the score as a small indicator next to each citation in the report. Flag any report with a low average credibility score.
03Authentication

Custom Auth: Still Carrying the Same Three Security Debts

The previous critique flagged three specific security issues introduced by replacing Clerk's hosted UI with custom auth forms. None of these appear to have been addressed in this revision. They remain P0 — the kind of issue that, if exploited, ends a company's trust relationship with its customers overnight.

P0 Not Yet Addressed
Custom auth forms running on your origin + no Content Security Policy + no explicit rate limiting on auth routes = three open doors. These are not hypothetical — credential stuffing attacks run continuously against every web application on the internet. The question isn't if, it's when.
P0 · Unresolved XSS on Your Origin = Credential Theft. Add CSP Headers.
Clerk's hosted UI ran on accounts.clerk.com — a separate origin. Any XSS on your Next.js app could not reach it. Your custom form runs on your domain. Any stored XSS anywhere on your platform is now also a login credential harvesting vector. This is the attack pattern used against Twilio, Okta, and multiple SaaS platforms in the last 3 years.
Fix today → Add Content-Security-Policy: script-src 'self'; object-src 'none' to your Next.js headers config. This blocks inline script injection. Then do a one-day XSS sweep of every rich text input, embed field, and user-facing content area in the platform. The HMAC webhook hardening you've already done proves you can implement this — apply the same rigour here.
P0 · Unresolved No Rate Limiting on Auth Routes
Clerk's hosted UI rate-limited login attempts server-side, on their infrastructure. Your custom form's underlying Clerk API calls go through your application layer first. Without explicit rate limiting on your auth routes, a credential stuffing script can attempt thousands of logins per minute against your application. Your Redis stack is already in place for this. It's a two-hour fix.
Fix today → Use Upstash Redis's built-in rate limiter: 5 failed auth attempts per IP per 15 minutes, then a 15-minute block. Add Cloudflare Turnstile (free CAPTCHA) that activates after 3 failed attempts. These two controls stop 99% of automated attacks. Upstash provides a code example for this exact use case in their docs.
Confirmed Strength HMAC Webhook Hardening is Properly Done
Strict HMAC verification on Clerk sync webhooks with resilient custom redirect logic — this is correctly implemented and worth calling out explicitly. Most developers either skip HMAC entirely or verify it incorrectly (comparing strings without constant-time comparison, which opens timing oracle attacks). Your implementation demonstrates the security-first thinking that the auth forms need applied to them.
04Multi-Tenancy

Org Content Scoping: The Right Change, One Refinement Needed

Strength New Members Instantly Seeing Team Projects is Table-Stakes Done Right
Refactoring tRPC routers to scope by organizationId so new team members inherit shared pipeline visibility on day one solves one of the most common B2B SaaS onboarding failures — the empty state problem where a new employee logs in and sees nothing. This single change dramatically improves the activation rate for team seats. Combined with the existing org presence engine and live activity feed, new members can immediately see what their team is working on. This is a product quality upgrade, not just an infrastructure one.
P1 · Data Model Day Plans Must Default to Private, Not Org-Scoped
The org-scoping refactor covers task and project data — which is correct, these are team-owned objects. But if the same scoping logic reaches DayPlan records, it creates a significant product problem. A daily agenda is a deeply personal document. If a team member's day plan is visible to their manager by default, they will stop using it honestly — or stop using it at all. The energy tracking (Sleep Quality, Stress, Mood) would be especially damaging to expose at the org level.
Verify → Confirm that DayPlan, EnergyLog, and WellbeingEntry records are scoped by userId only, not organizationId. If they've been caught in the org-scoping refactor, add a visibility: PRIVATE | SHARED | ORG enum field with PRIVATE as the default. Personal productivity data must never be org-visible without explicit user consent.
05UI/UX

The Design Investment is Paying Off. One Warning.

Strength Unified Dark-Mode Premium Aesthetic is a Competitive Signal
Overhauling Dashboard, Strategy, Canvas, Calendar, and Tasks to a unified high-end dark-mode aesthetic is not just cosmetic — it is a brand statement that positions you above Notion, ClickUp, and Monday.com in the premium segment. Linear built an entire company identity around this exact principle. Vercel did the same. When a product looks like it was built by people who care about craft, it attracts users who also care about craft. That is your target customer. Design quality is a retention strategy.
Strength Task Sync Across All Dashboard Views is a Felt-Quality Win
Ensuring task status changes (start, complete, pause) sync instantly across global dashboard views solves a subtle but significant trust problem. When a user completes a task in one view and sees it still showing as incomplete in another, they lose confidence in the data. This is the kind of "obviously it should work this way" bug that causes quiet churn — users don't complain, they just open Linear instead. Getting this right is more important than most new features.
P2 · UX Debt The Agenda/Backlog Layout Still Needs an Onboarding Moment
The Agenda vs. Backlog split-pane is the right UX decision — confirmed by how Sunsama, Morgen, and Linear approach daily planning. But this paradigm is unfamiliar to most users coming from task lists or kanban boards. Without a brief contextual explanation of what Agenda (committed today) vs. Backlog (available) means, first-time users will populate both the same way and lose the productivity benefit the layout was designed to create.
Fix → Add a 3-line tooltip on first use: "Agenda = what you're committed to doing today. Backlog = everything available to pull in." Consider an empty-state illustration that shows a half-full Agenda and a longer Backlog — it communicates the intended behaviour at a glance without any text at all.
06Persistent Gaps

Four Things That Have Now Appeared in All Four Reviews

Some items were flagged in Round 1 and remain unfixed through Round 4. At this point they are not oversights — they are prioritisation decisions. Here's why each one should move to the top of the queue.

4 Reviews. Still Missing. Optimistic UI — The Single Biggest Felt-Quality Gap
Every mutation — completing a task, moving a deal stage, updating a project status — waits for a server round trip before the UI reflects the change. On a standard connection this is 150–400ms of visible lag. Linear, Notion, and every top-tier productivity tool eliminate this with optimistic updates. The user action is reflected immediately in the UI; the server confirms in the background; rollback happens on error. The result is a product that feels native and instant vs. one that feels like a web form.
Fix → TanStack Query (already in your stack via tRPC) has onMutate for optimistic cache updates and onError for rollback. Start with the two highest-frequency mutations: task completion and deal stage change. One afternoon. The felt-quality improvement is disproportionate to the effort.
4 Reviews. Still Missing. Multi-Agent Orchestration on the Research Engine
You now have the best research UX in the market (Bloomberg terminal + streaming + grounded citations + full-page viewer). The architecture underneath it is still a single sequential agent. The gap between what users see and what the engine actually does will close when a competitor copies your UI. The architecture gap is your real moat. It's also the single largest determinant of report quality and speed.
Fix → Add an Orchestrator Inngest step before the research job. It calls Gemini with the query and returns a JSON research plan (3–6 sub-questions). Each sub-question becomes a parallel Inngest job using the existing search-grounded Gemini agent you've already built. Each returns structured findings. The Lead agent synthesises. Your existing streaming infrastructure handles real-time updates from all parallel jobs simultaneously.
2 Reviews. Still Missing. No Mobile / PWA for Daily Planning
Day planning and wellbeing logging — the two highest-frequency personal use cases — are things users want to do on their phone before the day starts. Without a mobile experience, your platform competes with Linear and Notion in a browser tab at a desk, which is their strongest context. On mobile, at 7am before the workday, there's no direct competition. This is retention-critical for the planning and wellbeing features.
Fix → Add a Next.js PWA manifest and service worker. Build a mobile-optimised version of /planning/day — large touch targets, swipe-to-complete tasks, one-tap energy logging. The data model is already correct. This is purely a responsive layout and PWA config task. 3–5 days of focused work closes this gap entirely.
2 Reviews. Still Missing. Collaborative Canvas Cursors — Your Demo is 80% Built
You have WebSocket presence, a canvas, org rooms, and user identity. Collaborative cursor rendering is approximately 40 lines of code away. The reason this matters is not the engineering — it's the sales demo. In an enterprise trial, the moment a prospect sees another cursor move on their canvas, the trial converts to a seat expansion conversation. Figma, Miro, and Linear all attribute significant activation to this single feature. You have every dependency in place.
Fix → On mousemove in the canvas, throttle to 50ms and emit {"{"}userId, name, color, x, y{"}"} to the org WebSocket room. Render incoming cursors as floating name tags positioned via the same CSS transform matrix as the canvas nodes. The quadtree handles culling. Total implementation: half a day.
